# Phase 3.5 — i18n Foundation + Retrofit Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a real internationalization framework (English + Indonesian) to `apps/next-web` and retrofit existing hardcoded-English UI to use it, so all current and future UI strings (including Phase 3.5 collaboration features) are translatable.

**Architecture:** Use `next-intl` in **"without i18n routing"** mode — the locale is read from a cookie (`pf_locale`), defaulting to `en`, NOT from a URL path segment. A server function resolves the active locale + messages; a client provider (`NextIntlClientProvider`) makes them available to client components via the `useTranslations` hook, while server components use `getTranslations`. Message catalogs live in `apps/next-web/messages/{en,id}.json` as nested namespaces. The retrofit is mechanical: each surface's hardcoded strings move into a namespace and are replaced with `t('key')` calls; EN values are the existing literals, ID values are translated.

**Tech Stack:** Next.js 16 (App Router), React 19, TypeScript, `next-intl`.

> ⚠️ **Next.js 16 caveat (from `apps/next-web/AGENTS.md`):** This is NOT the Next.js you may know — APIs and conventions differ from older versions. Before writing any framework-touching code (layout, providers, cookies, middleware), read the relevant guide under `apps/next-web/node_modules/next/dist/docs/`. In particular, verify the installed `next-intl` major version supports Next 16 App Router and "without i18n routing" mode; if the App Router API differs from the snippets here, adapt to the installed version's docs (the *structure* of this plan — cookie-driven locale, server resolver, client provider, namespaced messages — stays valid).

> **Why this plan exists / scope note:** Phase 3.5's design spec (`docs/superpowers/specs/2026-06-05-collaboration-gaps-phase3.5-design.md` §8) requires "i18n (EN + ID) for all new UI strings," but the codebase had **no i18n framework** — every existing string was hardcoded English. The user elected **full framework + retrofit**. This is a large, independent subsystem, so it is its own plan. **Tasks 1–6 (foundation) are a hard prerequisite for the Phase 3.5a/c collaboration UI tasks.** Tasks 7+ (retrofit of existing UI) are independent and may proceed in parallel with, or after, the collaboration slices.

```typescript
/**
 * Supported UI locales. `en` is the source-of-truth catalog; `id` mirrors it.
 * Locale is chosen via the `pf_locale` cookie (no URL-based routing).
 */
export const SUPPORTED_LOCALES = ['en', 'id'] as const;
export type AppLocale = (typeof SUPPORTED_LOCALES)[number];
export const DEFAULT_LOCALE: AppLocale = 'en';

/** Normalize an arbitrary cookie/header value to a supported locale. */
export function normalizeLocale(value: string | null | undefined): AppLocale {
  if (!value) return DEFAULT_LOCALE;
  const base = value.toLowerCase().split('-')[0];
  return (SUPPORTED_LOCALES as readonly string[]).includes(base)
    ? (base as AppLocale)
    : DEFAULT_LOCALE;
}
```

```typescript
import { describe, it, expect } from 'vitest';
import en from '../../../messages/en.json';
import id from '../../../messages/id.json';

/** Flatten nested message object to dotted key paths. */
function keys(obj: Record<string, any>, prefix = ''): string[] {
  return Object.entries(obj).flatMap(([k, v]) =>
    v && typeof v === 'object' ? keys(v, `${prefix}${k}.`) : [`${prefix}${k}`],
  );
}

describe('message catalogs', () => {
  it('en and id have identical key sets (no missing/extra translations)', () => {
    const enKeys = keys(en).sort();
    const idKeys = keys(id).sort();
    const missingInId = enKeys.filter((k) => !idKeys.includes(k));
    const extraInId = idKeys.filter((k) => !enKeys.includes(k));
    expect(missingInId, `keys missing from id.json: ${missingInId.join(', ')}`).toEqual([]);
    expect(extraInId, `keys present only in id.json: ${extraInId.join(', ')}`).toEqual([]);
  });

  it('has no empty string values', () => {
    const empties = keys(en).filter((k) => {
      const val = k.split('.').reduce<any>((o, part) => o?.[part], en);
      return val === '';
    });
    expect(empties).toEqual([]);
  });
});
```

```typescript
import { cookies } from 'next/headers';
import { getRequestConfig } from 'next-intl/server';
import { COOKIE } from '../server/cookies';
import { normalizeLocale } from './locale';
import { loadMessages } from './messages';

/**
 * Resolves the active locale per request from the `pf_locale` cookie
 * (no URL-based routing) and loads the matching message catalog.
 */
export default getRequestConfig(async () => {
  const cookieStore = await cookies();
  const locale = normalizeLocale(cookieStore.get(COOKIE.locale)?.value);
  return {
    locale,
    messages: await loadMessages(locale),
  };
});
```

- [ ] **Step 1: Create the locale-switch server action**

Create `apps/next-web/src/server/actions/locale.ts`:

```typescript
'use server';

import { cookies } from 'next/headers';
import { revalidatePath } from 'next/cache';
import { COOKIE } from '../cookies';
import { normalizeLocale, type AppLocale } from '@/i18n/locale';

/** Persist the user's UI language choice in the `pf_locale` cookie. */
export async function setLocale(next: AppLocale): Promise<void> {
  const locale = normalizeLocale(next);
  const store = await cookies();
  store.set(COOKIE.locale, locale, {
    path: '/',
    maxAge: 60 * 60 * 24 * 365, // 1 year
    sameSite: 'lax',
    httpOnly: false,
  });
  // Re-render server components with the new locale's messages.
  revalidatePath('/', 'layout');
}
```

## Retrofit (Tasks 7+) — convert existing hardcoded UI to i18n

> **This is the large, independent track.** It does not block the collaboration slices except that those slices' UI consumes the foundation from Tasks 1–6. Execute the retrofit surface-by-surface; each surface is one commit. The **key-parity test from Task 3** is your safety net — run it after every surface.

### The retrofit recipe (apply per surface)

For each file in the **Retrofit Checklist**:

1. **Identify** every user-visible literal string (button labels, headings, placeholders, empty states, aria-labels, toast messages, table headers). Skip: `className`, data keys, log strings, test ids, and developer-only strings.
2. **Choose a namespace** — the surface's PascalCase name (e.g. `Board`, `Backlog`, `Admin`, `ProjectSettings`). Reuse `Common` for generic words (Save/Cancel/Delete/Close/Edit/Loading).
3. **Add keys** to `messages/en.json` under that namespace — value = the existing literal verbatim.
4. **Add the same keys** to `messages/id.json` — value = the Indonesian translation.
5. **Consume in the component:**
   - Client component (`'use client'`): `const t = useTranslations('Namespace');` then replace `"Literal"` with `{t('key')}`.
   - Server component: `import { getTranslations } from 'next-intl/server';` then `const t = await getTranslations('Namespace');`.
   - Interpolation: `t('greeting', { name })` with `"greeting": "Hello {name}"`.
   - Pluralization: use ICU syntax, e.g. `"viewers": "{count, plural, =0 {No viewers} one {# viewer} other {# viewers}}"`.
6. **Run the parity test** (`npm run test:unit --workspace apps/next-web -- messages.unit`) — must pass.
7. **Visually verify** the surface in both locales (`npm run dev:web`, toggle language).
8. **Commit** one surface per commit: `git commit -m "i18n(retrofit): <surface>"`.
