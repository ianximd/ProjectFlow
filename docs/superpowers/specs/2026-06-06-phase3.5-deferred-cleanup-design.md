## Goals

Resolve, in one coordinated batch:

1. **Live board add/remove** — tasks created/moved/deleted by another user appear/move/disappear instantly, not only on the next SSR re-seed.
2. **SSE over-broadcast** — every authed client currently receives all `task:updated` events; scope delivery server-side.
3. **EVERYTHING views surface** — verify it is reachable and functional (already wired in Phase 3 follow-ups).
4. **i18n gap** — translate the table boolean `Yes/No` cell.
5. **Deprovisioned-author policy** — a user removed from the workspace must not be able to assign/resolve comments, including their own.
6. **Dependency advisories** — `hono` (moderate) and `next` (high).

Non-goals: per-workspace WorkflowStatus-name translation (status names remain user data); any change to the SSR-as-source-of-truth data model; per-user timezone infra.

## §1 + §2 — Full live board sync + per-project SSE scoping

These are one change: the vehicle that fixes over-broadcast (a keyed channel) is also what makes reliable live add/remove possible (full payloads on a scoped channel).

### Current state

- `apps/api/src/graphql/pubsub.ts`: `'task:updated': [{ projectId; task }]` — a **single, unkeyed** topic.
- Published from ~10 sites (`task.routes.ts` create/update/move; `schema.ts` createTask/updateTask/transitionTask/moveTask; a custom-field value-set path that emits a **partial** `{ task: { id } }`).
- `schema.ts` subscription `taskUpdated(projectId)`: `requireAuth` only, then `pubsub.subscribe('task:updated')` — **ignores `projectId`**, so every client gets every task event and scoping is purely client-side (`mergeTaskDelta` id-match).
- Client `useLiveTasks(projectId, base)` is **update-only**: a delta whose id isn't already shown is dropped (to avoid ghost cards from the partial publisher).

### Target design — server

**One keyed lifecycle topic** replaces `task:updated`:

```ts
'task:event': [scopeKey: string, {
  kind: 'created' | 'updated' | 'deleted';
  projectId: string;
  task?: unknown;     // full task for created/updated
  taskId?: string;    // for deleted
}];
```

**Central publish helper** `publishTaskEvent(kind, { task?, taskId?, projectId })` replaces all raw
`pubsub.publish('task:updated', …)` sites (removing the existing duplication). It publishes **twice**:

- key `prj:{projectId}` — consumed by scoped views (LIST/FOLDER/SPACE/board/lists).
- key `ws:{workspaceId}` — consumed by EVERYTHING views.

`workspaceId` is resolved from `projectId` (Space → Workspace) via a cached lookup (`withCache`, `TTL.LONG`,
same pattern as `userDisplay` in presence). Keys are **prefixed** (`prj:` / `ws:`) so a project GUID and a
workspace GUID can never collide on the same topic.

**Event mapping at publish sites:**

| Action | Event |
|---|---|
| create (mutation + REST) | `created`, full task |
| field update / transition / custom-field value-set | `updated`, full task (partial publisher still routes here; merge stays defensive) |
| move within same project | `updated`, full task (new `listId`/`status`) |
| move across projects | `deleted` on old `prj:` + `created` on new `prj:` |
| delete | `deleted`, `taskId` |

**Subscription** `taskEvents(projectId: String, workspaceId: String): TaskEvent!`:

- `projectId` provided → `requireObjectLevel(ctx, 'SPACE', projectId, 'VIEW')` → `subscribe('task:event', 'prj:'+projectId)`.
- else `workspaceId` provided (EVERYTHING) → `requireWorkspacePermission(ctx, workspaceId, 'workspace.read')` → `subscribe('task:event', 'ws:'+workspaceId)`.
- Exactly one of the two args is expected; neither → error.

This **eliminates the global channel** (closes §2) and **upgrades authz**: a client can only subscribe to a
scope it can VIEW (previously `requireAuth` only).

New GraphQL type `TaskEvent { kind: String!, task: Task, taskId: String }`. The old `taskUpdated` field is
removed (replaced by `taskEvents`).

### Target design — client

`useLiveTasks(base, { subscribeKey, accepts })`:

- `subscribeKey` = `{ projectId }` for scoped views or `{ workspaceId }` for EVERYTHING; drives the
  `taskEvents` variables. `skip` when neither is present.
- `accepts(task): boolean` decides whether a **created** task belongs in this view:
  - board (project) / space / EVERYTHING: accept all in scope (default `() => true`).
  - list view: `task.listId === currentListId`.
- `onData` switches on `kind`:
  - `updated` → existing `mergeTaskDelta(prev, delta)` (id-match, defensive merge — unchanged).
  - `created` → append the full task **iff** id not already present **and** `accepts(task)`; no ghost cards
    because the payload is full. Dedupe against the creator's own optimistic card by id.
  - `deleted` → remove by id (no-op if absent).
- The view's owning `projectId` is resolved **server-side** (LIST/FOLDER → owning Space id) and passed down
  to the view components, replacing the current `activeView.id` placeholder.

`mergeTaskDelta` and the `TaskDelta` shape are reused unchanged for the `updated` path. A small mapper turns a
`created` full-task payload into the client `Task` shape (reuse the existing normalize path).

### Files (indicative)

- API: `graphql/pubsub.ts` (topic), new `graphql/task-events.ts` (helper + Space→Workspace cache), `graphql/schema.ts` (subscription + `TaskEvent` type + publish-site swaps), `modules/tasks/task.routes.ts`, the custom-field value-set publish site.
- Web: `lib/realtime/operations.ts` (`TASK_EVENTS` sub), `lib/realtime/useLiveTasks.ts`, `lib/realtime/merge-task-delta.ts` (created mapper if needed), the view components (`board-view-engine`, `list-view`, `table-view`, `calendar-view`) + `board/board-view.tsx` + `lists/[listId]/list-view.tsx` + their pages (pass resolved `projectId`/`workspaceId` + `accepts`).

### Tests

- API unit: `publishTaskEvent` double-publish + key prefixing + kind mapping; Space→Workspace cache; subscription authz branch (SPACE VIEW vs workspace.read; reject when unauthorized).
- Web unit: `useLiveTasks` created/updated/deleted handling, `accepts` filtering, optimistic-dedupe, no-ghost on full created.
- Live e2e: extend `realtime-notifications`/a board spec — a second context creates/moves/deletes a task; the first sees it appear/move/disappear.

## §5 — Block deprovisioned authors from comment assign/resolve

`apps/api/src/graphql/schema.ts:39` `assertCanEditComment`:

```
requireAuth(ctx)
comment = getById(commentId)            // 404 if missing
task    = taskRepo.getById(comment.taskId)
workspaceId = resolve(task)             // via Space
assertWorkspaceMember(ctx.user, workspaceId)   // NEW — throws 403 if not a current member
if (ctx.user.userId === comment.authorId) return   // author (and member) may act
requireObjectLevel(ctx, 'LIST', listId, 'EDIT')
```

- The membership assertion runs **before** the author early-return, so a removed user can't act even on their
  own comments.
- Mirror the same guard on the REST `/comments/:id/assign` and `/resolve` paths (currently
  `requirePermission('task.update', { ownerFallback: comment.update.own })` — verify the ownerFallback does not
  let a non-member author through; add an explicit membership check if it does).
- Defense-in-depth: add an **actor-membership** check inside `usp_Comment_Assign` / `usp_Comment_Resolve`
  (throw → mapped to 403). The assign SP already validates the *assignee* is a member; this adds the *actor*.
- Locate the existing membership helper (`requireWorkspacePermission` / membership repo) rather than inventing one.

**Test:** integration — an author removed from the workspace attempting assign/resolve on their own comment is
rejected (both GraphQL and REST); an in-workspace author still succeeds.

## §6 — Dependency advisories

Two independent packages; do them as discrete commits so a problematic one can be isolated.

- **API / hono** (`<=4.12.20`, moderate; incl. "JWT middleware accepts any Authorization scheme" — relevant to
  the authed API): `npm audit fix` (in-range, non-breaking). Re-run API unit + integration.
- **Web / next** (`16.0.0–16.2.5`, high; App Router middleware/proxy bypass): `npm audit fix --force` → `next@16.2.7`.
  Because this app runs a customized Next ("this is NOT the Next.js you know"), **full verify**: tsc, `npm run build`,
  web unit, and the realtime/presence/views e2e. If the bump breaks anything non-trivial, **stop and report** —
  do not patch around it.
